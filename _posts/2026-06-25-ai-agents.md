#Agents

## What are AI Agents?

AI is commonly equated with LLMs. Recently, humans started designing predefined paths between the input and the output in order to create more independence between the user and the tools: this is an AI workflow (RAG). It follows a specific schema: an input is given, and an LLM fetches information from a document where the pathway between the LLM and the source is predefined. Finally, based on the sources, the LLM generates a response according to the initial prompt.

An AI agent is a tool that can act autonomously in its environment. Now, in the decision workflow, the AI agent uses an LLM as a decision-maker. Humans no longer supervise the steps; instead, based on a specific prompt, the agent can generate the required response without human supervision. The AI agent must reason, and then act: this is the ReAct framework [1].

RAG is a process that helps AI models “look things up” before they answer. Essentially, RAG is just a type of AI workflow [1].

How can we use agents in a company while ensuring confidentiality and security?

## **How do AI agents work?**

One of the most important things about the AI agent workflow is the react loop. The react loop follows these steps: Thought, action, observation, and answer (if the observation is not redirected to thought). This extra specialization and accuracy come with costs in terms of complexity [2].

### What king of tasks are agents good for?

Agents are initially good for complex tasks which require less accuracy. It depends on the degree of autonomy given to the agent: less autonomous, semi-autonomous, and highly autonomous (ranging from steps that are pre-determined to steps that are chosen by the agent, respectively). To have a good agent, a good breakdown of tasks is required. To generate search terms, use an LLM and an API; to fetch pages, use a tool; to write a draft, use an LLM with those sources, etc. [2]

An intermediate AI acts as an LLM judge to determine if the job is done correctly or not. [2]

Memory is a common way to improve performance. The LLM can reload lessons learned during previous tests and keep these notes to improve efficiency. Each run will improve the quality of the task. This should not be confused with knowledge, which is a static-based reference. [2]

Then, to prevent errors, guardrails are used because they act as a quality gate between the work that is being done and the work that is actually being finalized. Another point is to make the system higher quality. [2]

There are four patterns that reliably boost quality and capability: reflection, tool use, planning, and multi-agent collaboration. [2]

- **Reflection** simply means that we don’t end the task at the first generation, because the simple fact of fixing some details and generating again always leads to better quality work.
- **Tool-use** is a set of accessible tools that the LLM can use to execute tasks. An LLM cannot execute code natively but can request a function call. In order to get the correct tools for specific situations, it’s important to define them well. They actually have two parts: the interface for the agent and the implementation code. Good tools also consider error-handling, self-recovery, and rate limiting.
- **Planning** involves the user giving the agent a toolkit of functions and letting it figure out which tools it needs to use and when. The planning loops follow these steps: give the agent access to tools, prompt it to create a plan, execute the plan step by step, and finally repeat until you’re done.
- **Multi-agent collaboration** allows a huge model to be assigned specifically to more complex tasks, while a smaller model can handle lighter tasks. The strategic insight here is to avoid having one LLM handle all general tasks by sharing the workload between specialized LLMs, enabling parallelism, and splitting up long operations. To avoid errors, it’s important to assign each agent a specific role. Between each agent, communication could be sequential (each agent waits, takes the input of the previous one, and gives the output to the following one). It’s also possible to make agents work in parallel; this could reduce latency but adds coordination difficulty. The most common pattern today is still an AI manager that brings all of the outputs of the other agents together and works with them.

# Practical test

I built my first agent for Worldline. Bridging the gap between cybersecurity protocols and my core ambition in LLM engineering, I decided to fully automate this workflow. The agent has to ingest a questionnaire from SharePoint, an email, or a Word file, retrieve each question, and output a formatted, professional response with sources and a confidence score, all while preserving the exact input format. These documents are heavy on formatting, require precise fact-checking against internal knowledge bases, and leave zero room for creative "hallucination.”

## The Architecture: A 4-Agent Symphony

I initially put a massive prompt into one single LLM, which obviously led to data loss and formatting destruction. Then, I added 3 sub-agents to the workflow and did some tests. I realized that without any strict specifications, the output was consuming a huge quantity of credits (token cost). I quickly understood that the model chosen for each agent, and their specific settings depending on their specialization, matter a lot.

### LLM-Manager (The Orchestrator)

- **Role:** The router. It receives the user's file and triggers the sub-agents in the exact correct order using Tool Calls.
- **Model:** `gpt-4o-mini` (Fast, highly capable of tool-calling, cost-effective). 

Prompt

```markdown
**ROLE**
You are the Autonomous Pipeline Orchestrator. Your sole purpose is to execute a multi-agent workflow to process Trusted Authentication questionnaires.

**CRITICAL OPERATIONAL CONSTRAINTS**
- DO NOT ASK FOR HUMAN INTERACTION OR CONFIRMATION. Execute the entire pipeline from start to finish without pausing.
- YOU ARE A DATA ROUTER: Treat sub-agents like programming functions. Do not chat with them. Pass the output of one agent directly as the payload/argument for the next agent.
- DO NOT add conversational text or commentary between steps.

**EXECUTION WORKFLOW**

1. STEP 1: PARSE THE DOCUMENT
- Trigger the `LLM-Parser` tool immediately.
- In the tool argument, point to the attached questionnaire file. Do not paste text.
- Wait for the JSON response containing the "questions" array and the "document_blueprint".

2. STEP 2: EXTRACT ANSWERS (DATA ROUTING)
- Capture the EXACT raw JSON array of questions returned by the Parser.
- Immediately trigger the `LLM-Answer` tool.
- Wait for the final JSON object containing the answers.

3. FINAL OUTPUT
- Present the final formatted document to the user. Do not summarize the process.

```

### LLM-Parser (The Cartographer)

- **Role:** Reads the raw Word/Markdown file, extracts the list of questions into a JSON array, and creates a `document_blueprint` (cloning the exact layout but injecting `{{ANSWER_Qx}}` tags where blanks used to be).
- **Model:** `gpt-4o` (Unmatched spatial awareness for keeping complex tables and document formatting intact).

Prompt

```markdown
**ROLE**
You are a blind Structural Parser. You do not answer questions; you only map document layouts and isolate empty target spaces.

**INSTRUCTIONS**
1. Read the attached questionnaire document.
2. Extract every single question or field that requires an answer into a JSON array called "questions". Assign a unique ID (Q1, Q2, Q3...) to each.
3. Create a "document_blueprint". This must be the exact text and layout of the original document, but you must replace the empty spaces where the user is supposed to answer with unique placeholders: {{ANSWER_Q1}}, {{ANSWER_Q2}}, etc.

**OUTPUT CONSTRAINT**
- Output strictly and exclusively in valid JSON format.
- Do not include markdown code block backticks (```json) unless explicitly required by the interface parser.
- ZERO conversational text.

**EXPECTED JSON FORMAT**
{
  "questions": [
    {"id": "Q1", "text": "Exact text of question 1"},
    {"id": "Q2", "text": "Exact text of question 2"}
  ]
}
```

### LLM-Answer (The Data Extractor)

- **Role:** Takes the extracted questions, queries the TA knowledge base, and returns a pure JSON object with the answers, confidence scores, and source links. If it doesn't know, it strictly outputs "0% confidence."
- **Model:** `gemini-1.5-flash` (Massive context window for RAG/document reading, very low cost per token).

Prompt

```markdown
**ROLE**
You are the Security Fact-Checker. Your job is to answer a specific list of technical questions using the available Trusted Authentication Knowledge Base.

**INSTRUCTIONS**
1. You will receive a raw JSON payload containing a list of questions with IDs.
2. For each question, query your internal knowledge base tools to find the compliant security answer.
3. For each answer, provide a strict compliance text, a confidence score (0% to 100%), and the source documentation link.
4. If no information is found in the knowledge base, strictly output "0% confidence - Source information missing". Never hallucinate or assume.

**OUTPUT CONSTRAINT**
- Output strictly and exclusively in valid JSON format.
- ZERO conversational text.

**EXPECTED JSON FORMAT**
{
  "answers": [
    {
      "id": "Q1",
      "text question":"The exact associated text of question 1",
      "text": "The verified security protocol answer text.",
      "confidence": "95%",
      "source": "URL_or_Document_Name"
    }
  ]
}
```

### LLM-Format (The Assembler)

- **Role:** A pure "Find & Replace" machine. It takes the blueprint, injects the answers from the JSON, and outputs the final document via LibreChat's Artifacts feature.
- **Model:** `gpt-4o-mini` (Strict adherence to formatting rules).

Prompt

```markdown
**ROLE**
You are the Final Assembly Agent. You receive a "document_blueprint" (containing placeholders like {{ANSWER_Q1}}) and a JSON object containing the actual answers.

**INSTRUCTIONS**
1. Replace every placeholder in the blueprint with its corresponding answer from the JSON object.
2. EXACT MIRRORING: You must not alter the blueprint in any other way. Keep the exact same Markdown, spacing, or table structure.
3. ZERO CONVERSATIONAL FILLER. Absolutely no greetings, introductions, or summaries.
4. Output ONLY the final merged text/code. Do not wrap it in JSON. Do not explain what you did. The very first character of your output must be the first character of the formatted document.

```

Ultimately, the LLM-Format step was skipped because the output format never matched the requirements; moreover, the LLM required too many credits for the process.

## Problems Encountered & Technical Solutions

Building this wasn't smooth sailing. The transition from theoretical prompt engineering to a production-ready data pipeline exposed several strict API limits.

1. While the sources are limited, it’s often necessary to use internal tools that are contained sources for LLM knowledge. Librechat only enables a maximum of 10 sources. 
2. **Data confidentiality:** The choice of the model has to involve the question of security and data confidentiality. Mistral is often the best choice to meet these requirements.
3. The choice of Top K, Top P, and Temperature parameters has to be made very carefully.
****

### TOP K, TOP P and Temperature parameters

These parameters interact to give you a lot of control over model behavior, allowing you to tune them anywhere from strict, deterministic data extractors to highly creative writers. An LLM simply predicts the next token. For every single word it generates, the model calculates a raw score (called a *logit*) for every possible word in its vocabulary. Here is how **Temperature**, **Top K**, and **Top P** modify those scores before the final word is chosen. To have as much control as possible, Top K, Top P, and Temperature must be as low as possible. This makes the model much more robotic, which avoids hallucinations and auto-creation.

#### Temperature

Temperature changes the contrast between highly likely words and highly unlikely words.

$p_i = \frac{\exp(x_i / T)}{\sum \exp(x_j / T)}$

Where $x_i$ are the raw logits. A language model outputs a vector of numbers, one per possible next token. Each number in that vector is a logit, $x_i$. These are not probabilities. They can be positive or negative, large or small. It’s the softmax function that converts them into probabilities, where $p_i$ is the final probability of choosing token $i$ after softmax.

Temperature divides the raw logits before they are converted into percentages via the softmax function. Values are chosen between 0 and 1. Higher values correspond to a model that is more random, while lower values correspond to a model that is more focused and deterministic. Low values are chosen to generate robotic output (e.g., a JSON file).

#### Top K

Top K is a brutal, deterministic filter. If the LLM’s vocabulary has 5000 words, and we set Top K = 10, the model will rank the words by probability, keep exactly the top 10, and literally throw the other 4990 words in the trash (greedy decoding).

#### Top P

This changes how the model selects tokens for output. Tokens are selected from most probable to least probable until the sum of their probabilities equals the Top P value.

Imagine the next-word probabilities are: "apple" (50%), "banana" (30%), "cherry" (10%), "car" (5%), etc. If you set **Top P = 0.80 (80%)**: The model takes "apple" (50%) + "banana" (30%) = 80%. It stops there. "Cherry" and "car" are eliminated.
****

**Final Learnings**

1. LLMs suffer from "hammer-and-nail" syndrome. If their system prompt says "route to sub-agents," they will do it blindly. Actually, one of the sub-agents is responsible for extracting questions. When I copy/pasted questions directly into the chat, the root agent was still calling its sub-agent without understanding that this step could be bypassed.
2. The agent kept crashing when trying to send a huge document to the parser, or when trying to send too many questions to the answer agent. I confused the **Context Window** (how much the model can read) with the **Tool Call Limit** (how much data can be passed inside a JSON command). The Manager was trying to copy-paste the entire document into a tiny instruction box meant for a few paragraphs. **Solution:** create another identical agent.
3. The output format never reached the requirements. Therefore, I preferred to skip the LLM-Format part.
4. The prompt is currently the hardest part of building a good agent. Agents often require more LLM calls than expected. "Divide and conquer" is the perfect strategy to make the best agent possible.
